## Database Recommendation

For a healthcare startup building a patient management system, I would recommend using MySQL as the primary database. Healthcare environments demand strict data integrity and consistency; a patient's blood type or surgical history cannot afford to be "eventually consistent." MySQL provides ACID (Atomicity, Consistency, Isolation, Durability) compliance, ensuring that every transaction—such as updating a prescription—is either fully completed or rolled back, leaving the database in a valid state.

According to the CAP theorem, a distributed system can only guarantee two of three properties: Consistency, Availability, and Partition tolerance. In healthcare, we must prioritize Consistency and Partition tolerance (CP). While high Availability is desirable, it is better for a system to be temporarily offline than to show a doctor the wrong medication dosage due to a synchronization lag. MongoDB, by default, follows the BASE model (Basically Available, Soft state, Eventual consistency), which is less suited for core medical records but excellent for horizontal scaling.

However, if the startup needs to add a fraud detection module, the recommendation shifts toward a polyglot persistence architecture. Fraud detection involves analyzing massive volumes of semi-structured data, such as login locations, device IDs, and rapid-fire payment attempts, in real-time. MongoDB excels here because of its "Schema-on-Read" flexibility and ability to ingest high-velocity data without the overhead of complex joins. I would recommend a hybrid approach: MySQL for the mission-critical Patient Information (the system of record) and MongoDB for the Fraud Detection and Analytics layer. This allows the startup to maintain 100% data accuracy for patients while benefiting from the high-speed ingestion and horizontal scalability required for modern anomaly detection.
